## 什么是 Vibe Coding？

Vibe coding（氛围编程）是一种开发方式：你向 AI 编程助手描述意图，迭代其输出——而非逐行手写代码。人类负责架构、决策与质量判断，AI 负责实现、样板代码、调试与一致性维护。

这不是"让 AI 写应用"。这是一种新型结对编程：其中一个伙伴永不疲倦、从不忘记语法、能在数秒内重写整个模块——但需要清晰的方向和主动的监督才能保持正轨。

## B1 — Vibe Coding 实际的感受

### 转变

传统编码：你知道要写什么，只需要写出来。
Vibe coding：你知道自己想要什么，描述它，然后判断输出。

瓶颈从*打字*转移到*清晰表达*。你越能清楚地说明自己想要什么——包括约束、边界情况以及不应该做什么——结果就越好。

### 人类仍然要做的事

- 拥有架构和重大决策
- 在提交前审阅每一个 diff
- 发现 AI 偏轨或过度工程化的时机
- 即使输出看起来正确，也能知道它是否错了
- 管理 git、部署、外部账号

### AI 擅长的事

- 精确且完整地实现规格
- 同时在多个文件中保持一致性
- 记住代码库中的每个函数签名和类型
- 重构而不破坏接口
- 为刚写的代码编写测试

### AI 不擅长的事

- 假设它没有的上下文（过时的记忆）
- 在简单就够的情况下倾向于复杂性
- 无法判断"够用"与"完美"的区别
- 添加你没要求的功能和注释
- 在对话后期忘记早期的约束

### 技巧

- 给出明确的负面约束：*"不要为不可能发生的情况添加错误处理"*
- 说你想要什么，而不是怎么做——让 AI 选择实现方式
- 审阅 diff，而不只是最终文件——偏差藏在中间
- 简短的聚焦请求优于冗长的多部分请求

## B2 — 从架构文档开始，而不是代码

### 错误

直觉是立刻开始编码。有了 AI 助手，你*可以*——但你会花两倍时间修复结果。AI 需要一张地图，没有地图，它会自己发明一张。

### 实践方法

先写 `ARCHITECTURE.md`，定义：
- 模块边界（每个部分负责什么）
- 模块之间的数据契约（JSON 数据结构）
- 技术选型及其理由
- 明确的范围外内容

当 AI 拥有这份文档时，每个实现决策都有参考点。没有它，你会得到看似合理却无法组合在一起的代码。

### FIRE51 的实践

在写第一行服务器代码之前，我们定义了：
- 五个模块（输入 → 引擎 → 输出 → 建议 → 策略测试器）
- JSON 接口契约（`interface_m1_m2.json`、`interface_m2_m3.json`）
- 年度循环顺序（收入 → 支出 → RMD → 转换 → 税务 → 提款 → 增长 → 标记）

然后引擎严格按照规格实现，集成时没有任何意外。

### 技巧

- 在修改代码*之前*更新架构文档，而不是之后
- 用文档说不："根据 ARCHITECTURE.md §3，这超出范围"
- 当 AI 提出与文档不一致的内容时，尽早发现

## B3 — PDF 生成地狱：7 个问题与解决方案

从网页报告生成 PDF 听起来很简单，实际上它的迭代次数比其他任何功能都多。

| # | 问题 | 根本原因 | 解决方案 |
|---|------|---------|---------|
| 1 | Puppeteer 在 512MB 服务器上 OOM 崩溃 | Chromium 会启动完整的浏览器进程 | `--single-process --no-sandbox`；本地构建，rsync dist/ |
| 2 | CDN 脚本（Chart.js）无法加载 | `page.setContent()` 不执行外部请求 | 改用 `page.goto(file://)` |
| 3 | PDF 中图表空白 | Chart.js 动画是异步的，Puppeteer 截图太早 | `?pdf=1` 参数 → `Chart.defaults.animation = false` |
| 4 | 水印被白色背景遮住 | `@media print` 重置导致所有单元格不透明 | 半透明 `rgba` 背景；显式 body 覆盖 |
| 5 | iOS `window.print()` 无声失败 | WKWebView 未实现 print | 三路导出：原生 → 服务端 Puppeteer → 引导提示 |
| 6 | Capacitor 内置浏览器中 JWT 丢失 | WKWebView 上下文不共享 `localStorage` | 以 URL 查询参数读取 JWT 作为后备 |
| 7 | PDF 在渲染完成前就截图 | 图表 + 表格异步绘制 | `window.__reportReady = true` 哨兵；`waitForFunction` |

### 经验教训

通过无头浏览器生成服务端 PDF 是跨平台一致性的正确架构，但它需要与页面渲染生命周期进行显式同步——浏览器不会告诉你 JavaScript 什么时候绘制完毕。

## B4 — Emoji 搞垮了我的 iOS 应用

### 问题

图标在 Chrome 中正常显示，在 iOS 模拟器中也正常。但在真实 iOS 设备上：`[?][?][?]`。

如果字体不包含这些字符，iOS WebView 上任何超出基本拉丁字符范围的 Unicode 符号都有风险，包括常见符号，如 ☰（汉堡菜单）和 ⬆（箭头）。

### 四轮修复

| 轮次 | 方法 | 结果 |
|------|------|------|
| 1 | 添加 `system-ui` 字体后备 | 修复了部分，不是全部 |
| 2 | 替换为 HTML 实体 | 某些字形仍然失效 |
| 3 | 替换为 ASCII 替代方案 | 部分有效 |
| 4 | 使用明确字体大小/颜色的 CSS `::before` 伪元素 | 全部修复 |

### 规则

**永远不要在面向 iOS WebView 的 Web 应用中使用 Emoji 或 Unicode 符号。**
使用 CSS 伪元素、SVG 图标或图标字体（带后备）。

### 经验教训

在真实设备上测试，而不只是模拟器。模拟器使用 macOS 字体，设备使用 iOS 字体，它们不同。

## B5 — 让 AI 保持诚实：锁文件、测试与设计文档

### 问题

AI 编程助手既有帮助又很顺从，太顺从了。如果不加约束，它会：
- 修改测试数据以通过测试
- 简化输入以规避边界情况
- 重构你没让它碰的代码
- 添加你没有要求的功能

### 我们的做法

**明确锁定稳定文件。** 在 `CLAUDE.md` 中：
> `midupperclass2M_baseline.json` 和 `upperclass5M_baseline.json` 已锁定。
> 未经明确用户指令，不得修改数据。

这一条指令防止了 AI "修复"测试失败时通过更改输入而不是代码来解决问题。

**使用真实的验证套件。** 我们使用 PolicyEngine（一个独立的税务计算器）来验证税务引擎输出。AI 无法欺骗外部事实来源。

**保持输出格式稳定。** 同样在 `CLAUDE.md` 中：
> `run_simulation.ts` 的输出格式在 V1 中已锁定。未经明确用户指令，不得修改。

**频繁提交。** 每次提交都是一个检查点。如果 AI 偏轨，你可以在 diff 中精确看到哪里出了问题。

### 技巧

- 将约束写入 `CLAUDE.md`——AI 每次会话都会读它
- 锁定任何 AI 可能会"顺手"改掉以让其他东西正常工作的内容
- 将测试失败视为真正的失败，而不是修改测试的邀请

## B6 — 从想法到上线，用天而非月

### 交付了什么

完整的全栈生产应用，包含：
- 聊天式财务数据收集
- 逐年模拟引擎（TypeScript，纯函数）
- 带图表的交互式报告、PDF 导出、CSV 下载
- 带 OTP 邮件的 JWT 认证
- Capacitor iOS/Android 壳
- PolicyEngine 税务验证套件
- AWS Lightsail 上的 Nginx + SSL + PM2 部署

### 时间线

| 周次 | 工作内容 |
|------|---------|
| 第 1 周 | 架构文档、引擎、基础报告 |
| 第 2 周 | 上线 fire51.com、安全加固、税务引擎精度 |
| 第 3 周 | JWT 认证、OTP、移动端壳、服务端 PDF |
| 第 4 周 | 打磨、Emoji 修复、合并到主分支、新 VM 部署 |

### 关键推动因素

- **设计文档优先** — 集成时没有意外
- **纯函数** — 引擎在服务器、Web 和移动端上完全相同地运行

### 诚实的核算

AI 写了约 95% 的代码行。人类撰写了架构、做了每一个重大决策、发现了每一个重大 Bug，并指导了每一次航向修正。这不是"AI 构建了它"，这是一种新型的工程杠杆。

## B7 — 税务引擎精度：用 PolicyEngine 验证 AI 写的代码

### 问题

AI 写的财务代码看起来合理，但合理不等于正确。一个带有 5% 税务误差的退休预测，在 45 年后会复利放大成一个毫无意义的数字。

### 方法

PolicyEngine 是政策研究人员使用的开源微观模拟模型。我们将其用作独立的事实来源，来验证税务引擎的输出。

**流水线：**
1. 运行 FIRE51 模拟 → 将关键年份的税务输入 + 输出捕获到 MySQL
2. `npm run validate:pe` → 批量调用 PolicyEngine，比较结果，标记异常值
3. `npm run validate:report` → 并排对比表

**容差：** 联邦税 ±$100，州税 ±$300，NIIT ±$50，SS 应税额 ±$50

**通过验证发现的 Bug：**
- NIIT MAGI 未包含资本利得（低估了大额股票提款的 NIIT）
- SS 暂定收入未包含资本利得（低估了 SS 的应税性）
- 65 岁以上额外标准扣除额缺失
- 加州 SS 豁免缺失

全部由验证器发现，而非通过阅读代码。

### 技巧

对于任何有外部事实来源的领域（税法、金融法规、物理学），都要对其进行验证。AI 的输出看起来会是正确的，但它可能不是。

## B8 — Vibe Coding 目前还做不到什么

### 诚实的局限性

| 局限性 | 会发生什么 | 解决方法 |
|--------|-----------|---------|
| **上下文过时** | 忘记之前会话中的决策 | 维护 `CLAUDE.md` + 记忆文件 |
| **坐标盲区** | 无法看到 UI 元素在屏幕上的位置 | Computer-use API（昂贵）；明确描述布局 |
| **无审美** | 无法判断 UX 是否好，只能判断是否有效 | 人类审阅每个屏幕 |
| **计费系统** | 无法访问 AWS 控制台、DNS、支付账号 | 人类处理所有外部账号工作 |
| **交互式终端** | 无法处理需要键盘输入的提示 | 传递 `--non-interactive` 标志；编写脚本绕过 |
| **规模成本** | UI 测试的 Computer-use：约 $0.50/次 | 已推迟；改用 `?test` 模式 |

### 不可简化的人类角色

- **愿景**：这个产品应该是什么？
- **判断**：这个输出实际上正确吗？
- **审美**：这对用户来说感觉对吗？
- **责任**：出错时谁负责？

这些不会消失，它们成为了整个工作本身。

## B9 — 代码与文档同步：三层强制机制

### 问题

在 vibe coding 中，AI 写代码很快，文档也会快速过时。几次会话后，设计文档描述的是一个已不存在的系统。新会话从一张过时的地图开始，决策被反复争论，Bug 重新出现。

根本原因：没有自然的强制机制。在团队中，代码审查能发现偏差；在单人 vibe coding 会话中，除非你刻意构建它，否则什么都不会。

### 三层机制

#### 第一层 — AI 常规指令（`CLAUDE.md`）

`CLAUDE.md` 在每次会话开始时被读取，是 AI 的持久指令集。将纪律规则写在那里，AI 就会自动应用它们——无需每次提醒。

FIRE51 的 `CLAUDE.md` 现在包含：
- **文档同步表**：将每个代码区域映射到必须随之更新的文档
- **VIBECODING.md 日记规则**：定义哪类观察属于原始笔记
- **提交前检查清单**：每次提交前需回答的 4 个问题
- **锁定文件列表**：AI 未经明确指令不得触碰的文件

这是杠杆最大的一层，提交时零成本，却塑造了每次会话。

```
| 代码改动范围                    | 需更新的文档               |
|---------------------------------|--------------------------|
| 引擎逻辑、税务、RMD             | MODULE2_DESIGN.md        |
| 认证、OTP、服务器路由           | GOLIVE_DESIGN.md         |
| 移动端 / Capacitor              | CAPACITOR_DESIGN.md      |
| 架构 / 模块边界                 | ARCHITECTURE.md          |
| 任何困难或出人意料的修复        | VIBECODING.md 原始笔记   |
```

#### 第二层 — Git 预提交钩子（人类提醒）

预提交钩子在每次 `git commit` 时触发，无法被遗忘。

FIRE51 的钩子：
- 检测到 `src/` 或 `public/` 文件被暂存但没有 `.md` 文件被暂存时打印可见的提醒框——*软警告，不阻止提交*
- 仅在 `TaxEngine.ts` 变更未通过 PolicyEngine 验证时硬性阻止

```
  ┌─────────────────────────────────────────────────────────┐
  │  文档同步提醒                                           │
  │                                                         │
  │  代码已变更但没有暂存的 .md 文件。                       │
  │  → 相关设计文档是否已更新？                              │
  │  → 是否有值得添加到 VIBECODING.md 原始笔记的内容？      │
  └─────────────────────────────────────────────────────────┘
```

设计为软性。对每次提交都硬性阻止会产生摩擦，最终被绕过。在正确时机的可见提醒已经足够。

#### 第三层 — 文化：写下来，而非口口相传

纪律记录在两个地方：
- `CLAUDE.md` — AI 读它
- `VIBECODING.md`（本文件）— 人类读它

当规则只存在于某人脑中时，一旦注意力转移就会消失。当规则被写下并提交到仓库中，它就能跨会话、跨协作者、跨数月的项目空窗期持续存在。

### 为什么软性执行优于硬性执行

硬性执行（没有文档更新就阻止提交）听起来很吸引人。实践中，它会产生被绕过的摩擦：`git commit --no-verify` 的存在是有原因的。

目标不是让违规变得不可能，而是让正确行为成为阻力最小的路径。在提交时的可见提醒，结合被指示在每次任务中更新文档的 AI，比一个门控系统更持久。

### 日记规则

每当一个问题需要多次尝试、有出人意料的根本原因，或产生了一个有价值的教训时——在 `VIBECODING.md` 原始笔记中添加一条记录。一行就够了。

门槛故意设得很低。这样的一条笔记：
> "setup.sh 需要 `set -a && source .env` 来加载 DB_PASS——手动 export 在 CI/部署中容易出错"

只需 10 秒写完，却价值一个博客段落。

原始笔记积累起来，博客文章自然就写成了。

### 实践中是什么样子

之前：会话结束，代码被提交，文档过时，教训被遗忘。

之后：
1. AI 作为同一任务的一部分更新相关设计文档
2. 预提交钩子触发——人类确认或添加后续说明
3. 如果问题有趣，一条记录进入原始笔记
4. 下次会话从准确的地图和不断增长的日记开始

每次会话的开销：不到 2 分钟。复利价值：每次未来的会话、每篇博客文章、每位新贡献者都从真实情况出发，而不是从考古学出发。

## B10 — Git 钩子：你的隐形副驾驶

### 是什么

Git 钩子是 git 在工作流特定节点自动运行的 shell 脚本，存放在 `.git/hooks/` 中，git 按名称调用它们——无需配置。

```
.git/hooks/
  pre-commit     ← 在提交创建前触发
  commit-msg     ← 在你输入提交信息后触发
  pre-push       ← 在 git 推送到远端前触发
  post-commit    ← 在提交完成后触发（无法中止）
```

当你运行 `git commit` 时，git 会查找 `.git/hooks/pre-commit`。如果它存在且可执行（`chmod +x`），git 就会运行它：
- 退出 `0` → 提交继续
- 退出非零 → 提交中止

### FIRE51 的使用方式

**`pre-commit`** — 在每次 `git commit` 时触发：
- 暂存了 `src/` 或 `public/` 文件但没有 `.md` 文件 → 打印文档同步提醒（软性，退出 `0`）
- `TaxEngine.ts` 被暂存 → 运行 PolicyEngine 验证 → 如果测试失败则硬性阻止（退出 `1`）

**`pre-push`** — 在每次 `git push` 前触发：
- `TaxEngine.ts` 相对于远端有变更 → 运行 PolicyEngine 验证 → 如果失败则阻止推送

两个不同的钩子，两个不同的时机。pre-commit 在问题被记录之前捕获，pre-push 是它们到达远端之前的最后一道防线。

### 快速参考

| 钩子 | 时机 | 退出非零则 |
|------|------|-----------|
| `pre-commit` | 提交创建前 | 中止提交 |
| `commit-msg` | 信息写完后 | 中止提交 |
| `pre-push` | 推送到远端前 | 中止推送 |
| `post-commit` | 提交完成后 | （无法中止）|
| `pre-rebase` | rebase 开始前 | 中止 rebase |

### 关键限制：钩子不被提交

`.git/hooks/` 在设计上被排除在 git 跟踪之外。新的 `git clone` 会得到一个空的钩子文件夹，其他协作者不会自动获得你的钩子。

标准解决方法：
1. **`setup.sh` 复制它们** — 将钩子提交到 `scripts/hooks/`，让 setup.sh 创建符号链接或复制到 `.git/hooks/`
2. **`package.json` + husky** — 将钩子作为项目一部分管理的 npm 包
3. **文档化并信任** — 对于单人或小团队来说效果不错

FIRE51 目前使用方案 3，方案 1 是有协作者加入时的正确下一步。

### 软性与硬性执行

文档同步提醒是软性的（退出 `0`，从不阻止）。税务验证是硬性的（退出 `1`，阻止提交）。这是刻意的。

**硬性合适的情况：** 违规是客观上的破坏（测试失败、linting 错误、检测到密钥）。坏提交的代价足够高，摩擦是合理的。

**软性合适的情况：** 检查是关于实践的提醒，而非正确性门控。将文档更新做成硬性阻止会立刻产生 `git commit --no-verify` 习惯。在正确时机的可见提醒能改变行为，而不产生抵触。

### 在 vibe coding 中特别适合

钩子天然适合 vibe coding，因为 AI 动作很快。没有门控，不良习惯会跨会话积累。有了钩子，质量检查就在自然关注提交的那一刻发生——在它完成之前。

AI 也可以写钩子本身。在 FIRE51 中，预提交钩子由 Claude 编写、提交，并从此静静运行。零维护成本，持久的护栏。

**Cursor（免费）：** IDE 层。内联补全和快速文件导航。用于代理工作的频率低于 Claude Code，但在 Claude Code 处理另一个文件时有助于在代码库中保持方向感。

Cursor 的免费版有限制——每月消息上限，且无法访问最大的模型。对于以 vibe coding 速度前进的单人项目，免费版表现良好。被限制所阻碍的那一刻就是升级的时机；对 FIRE51 来说，那一刻尚未到来。

你不需要强大的机器、昂贵的 IDE 许可，或复杂的云设置。门槛是：

1. $20/月 用于一个有能力的 AI 代理（Claude Code 或同类产品）
2. $12/月 用于一台小型云服务器（或从免费 VPS 开始）
3. 一台能运行代码编辑器和终端的笔记本

其他一切——框架、运行时、数据库、Web 服务器——都是开源且免费的。

Vibe coding 的真实成本是**注意力，而不是金钱**。你需要阅读每一个 diff、验证每一个设计决策，并在整个过程中保持质量判断。AI 负责打字，你负责思考。这就是这笔交易，而且是一笔好买卖。

这不是为优化工程人力成本的公司写的教程。它是为那些在车库里构建梦想的人写的。

## B11 — 保持 CLAUDE.md 简短

### 长指令文件的问题

在 vibe coding 中，`CLAUDE.md` 在每次会话开始时被加载到 AI 的上下文中。它是你跨对话塑造 AI 行为最强大的杠杆——比记忆更可靠，比内联提示更持久。

但它只有在被实际阅读和应用时才有效，这要求它必须简短。

一个 200 行的 `CLAUDE.md` 是个陷阱。文件以正确的想法开始——纪律规则、锁定文件、提交前检查清单。然后，逐渐地，有人（或 AI）添加了项目背景、场景描述、架构图、设计说明。每次添加看似无害，合在一起，它们把重要的指令掩埋了。

AI 加载整个文件，但它在优化相关性。第 5 行的纪律规则比同样的规则在第 150 行——被它今天不需要行动的历史背景包围——权重更高。

### 规则：CLAUDE.md 是指令，不是文档

`CLAUDE.md` 中的所有内容都应该回答以下问题之一：
- 我必须始终做什么？（纪律规则、文档同步表）
- 我绝不能做什么？（锁定文件、禁用模式）
- 我现在需要知道什么才能写出正确的代码？（年度循环顺序、V1 简化）
- 我在哪里查找详情？（文档索引指针）

如果它不回答这些问题之一，它就属于设计文档，而不是 `CLAUDE.md`。

**移至设计文档：**
- 项目概述 → `FIRE51.md`
- 场景描述 → `NEXT_STEPS.md` 或 `ARCHITECTURE.md`
- 架构图 → `ARCHITECTURE.md`
- 已知简化 → `DEFERRED.md`
- 历史里程碑 → `NEXT_STEPS.md`

### 保留什么

FIRE51 的 `CLAUDE.md` 在精简后从 196 行减少到 69 行。留下的是：

1. **纪律规则** — 文档同步表、VIBECODING 日记规则、锁定文件、提交前检查清单。这些是 AI 在每次任务中必须应用的行为指令。

2. **关键技术事实** — 年度循环顺序、转换策略、提款优先级、增长率、仍在生效的 V1 简化。AI 写出*正确*的引擎代码所需的事实，无需先阅读其他三个文件。

3. **文档索引** — 每个领域对应正确文档的一行指针。保持 CLAUDE.md 精简，同时给 AI 一张在需要时深入获取上下文的地图。

### 好处

当 `CLAUDE.md` 只有 69 行时，纪律规则无法被忽视。AI 始终如一地应用它们，因为它们主导了指令上下文——不是因为它在尝试记住文本墙第 150 行的某些内容。

简短 = 响亮。最重要的指令应该是唯一的指令。

### 维护规则

向 `CLAUDE.md` 添加内容时，问自己：*这是 AI 必须遵循的规则，还是它可能觉得有用的上下文？* 规则属于这里，上下文属于它所描述的文档。

每当你注意到 AI 遗漏了本应知道的指令时，回顾 `CLAUDE.md` 的长度。臃肿通常是原因。
